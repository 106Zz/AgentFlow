# RAG 系统工程化三大核心痛点与解决方案

本文档记录了在构建企业级 RAG（检索增强生成）系统时遇到的三个关键工程化挑战及其解决方案。这些经验对比了主流 AI 产品（ChatGPT、Claude、豆包）的实现机制，旨在指导系统的架构优化。

## 1. 实时 Markdown 渲染 (Real-time Markdown Rendering)

### 痛点现象
在流式输出（Streaming）过程中，如果使用普通的 Markdown 解析器，当 Markdown 语法（如代码块 ` ``` `、加粗 `**`）尚未闭合时，界面会出现：
*   原始内容直接暴露（如显示 `**加粗文...`）。
*   布局剧烈抖动（解析器在“纯文本”和“HTML标签”之间反复跳变）。
*   代码块无法高亮或格式错乱。

### 误区纠正
*   **误区**：认为必须等待后端生成完整段落或完整闭合标签后才能渲染。
*   **真相**：主流产品使用**容错流式解析**策略。

### 解决方案
*   **流式补全 (Stream Auto-completion)**：前端解析器在接收到不完整的 Markdown 标记（如 ` ```java `）时，应在内存中自动补全闭合标签（` ``` `），生成临时的合法 HTML 进行渲染。
*   **CSS 容错**：使用 `white-space: pre-wrap` 确保即使解析失败，文本也能保留换行和空格格式。
*   **渲染频率控制**：不应按字符触发渲染（性能开销大且闪烁严重），应按**时间片（如 16ms/60fps）** 或 **数据块（Chunk）** 进行节流渲染。

## 2. 会话中断导致数据丢失 (Data Persistence on Interruption)

### 痛点现象
用户在回答生成的过程中刷新页面、关闭网页或切换到其他会话（即 Client Abort），导致：
*   **数据库无记录**：再次进入会话，刚才的问题和生成了一半的回答完全消失。
*   **后端报错**：Spring Boot 抛出 `IOException: Broken pipe`，导致事务回滚或保存逻辑未执行。

### 核心架构缺陷
目前的逻辑是：`HTTP 连接建立 -> 流式生成 -> (连接保持中) -> 生成结束 -> 写入数据库`。
一旦连接断开，后续的“写入数据库”步骤被中断。

### 解决方案 (Backend Architecture)

#### 方案 A：先落库，后生成 (Write-Ahead)
1.  用户请求到达时，**立即**在数据库插入一条 `UserMessage` 和一条内容为空（或占位符）的 `AssistantMessage`。
2.  无论后续生成是否成功，记录的“骨架”已经存在，用户再次刷新也能看到问题。

#### 方案 B：断点续存与异常捕获 (Save-on-Abort)
1.  **异常捕获**：在后端流式处理（如 WebFlux `doOnCancel` 或 MVC 异常拦截）中，专门捕获“客户端断开”异常。
2.  **强制刷盘**：捕获到断开时，**不回滚**，而是将 StringBuilder 中已生成的缓冲区内容 `UPDATE` 到数据库。
    *   *注*：这需要 AI 生成线程与 HTTP 响应线程适当解耦，防止 HTTP 线程死亡导致无法执行后续逻辑。

#### 方案 C：完全异步解耦 (Full Async - 进阶)
1.  前端请求仅触发任务，后端将任务推入**线程池**或**消息队列**。
2.  后端线程独立执行 RAG + 生成逻辑，并将结果实时写入 Redis 或数据库。
3.  前端通过 WebSocket 或 SSE 仅作为“观察者”订阅结果。即使观察者离线，生成任务依然在后台完成。

## 3. 侧边栏主题即时更新 (Instant Sidebar Updates)

### 痛点现象
用户发送第一条消息后，侧边栏的会话标题（通常是“新对话”）没有变化。必须手动刷新页面，或者等待整个回答生成完毕，标题才会更新。

### 解决方案 (Optimistic UI + Async)

#### 前端：乐观更新 (Optimistic UI)
*   **策略**：在用户点击“发送”的一瞬间（请求未发出前），前端直接截取用户输入的前 10-15 个字符。
*   **操作**：直接修改本地 Store 中当前会话的 `title` 属性。
*   **效果**：侧边栏标题瞬间变更，给用户极佳的响应感。

#### 后端：旁路异步生成
*   **策略**：不让“起标题”这个任务阻塞主对话流。
*   **操作**：
    1.  主线程处理 RAG 问答。
    2.  **同时**（或在首个 chunk 返回后）提交一个异步任务给小模型（如 Qwen-Turbo, GPT-3.5）。
    3.  任务内容：“请为这段对话生成一个简短的 4-6 字标题”。
    4.  生成完成后，默默更新数据库，并通过 WebSocket/SSE 推送给前端（或等用户下次加载时生效）。
